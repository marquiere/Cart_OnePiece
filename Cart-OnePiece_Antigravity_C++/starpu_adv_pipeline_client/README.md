# CArt-OnePiece Framework: StarPU ADV Pipeline Client

This folder contains the Advanced Autonomous Driving (ADV) workload measurement client for the CArt-OnePiece framework. It connects a CARLA 0.9.15 simulator to an asynchronous sensor processing pipeline accelerated by StarPU and TensorRT.

## Project Structure
- `apps/` - Main pipeline executables and smoke tests.
- `src/` - Core implementation source files (Synchronization, Preprocessing, Metrics).
- `include/` - Header files.
- `tools/` - Run scripts for servers, clients, profiling, and monitoring.
- `third_party/` - Dependencies like `stb_image` and tiny utilities.
- `models/` - Placeholder for TensorRT engine files (`.engine`).
- `runs/` - Autogenerated output folder for dataset dumps, logs, and CSV metrics.

---

## 1. Prerequisites & Building

**Dependencies:**
- **CARLA 0.9.15** (Simulator and C++ Client API)
- **StarPU 1.4+** (Task programming library)
- **TensorRT & CUDA** (For inference)
- **CMake 3.14+** and **C++17** compiler

**Building the Project:**
```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DWITH_TRT=ON -DWITH_STARPU=ON
cmake --build build -j
```
*Note: If `pkg-config` cannot find StarPU, ensure `PKG_CONFIG_PATH` is set. If CARLA headers are missing, verify `CARLA_ROOT` environment variable.*

## 2. Core Usage: Running the Simulation

The client requires an active CARLA server. You should always use the provided bash scripts to manage the server lifecycle and prevent zombie UE4 processes.

### Starting the Server
```bash
./tools/run_server.sh --port 2000
```
*(If `CARLA_ROOT` isn't in your bash profile, pass it manually: `--carla_root /path/to/carla`)*

Logs will be written to `runs/<timestamp>/server.log`.

### Stopping the Server
```bash
./tools/kill_server.sh
```

### Full Automated Evaluation (All-in-One)
To run the complete demonstration pipeline—which safely boots the CARLA server, waits for the map to load, extracts a visual sanity dataset, runs the profiled StarPU pipeline for 120 frames with background telemetry tracking, generates the workload summary reports, and cleans up the background server automatically—use the unified script:

```bash
# Basic run with the default Dummy Model generating a dataset and validating inference
./tools/run_pipeline.sh --frames 200 --run-dataset

# Advanced run sweeping through all schedulers
./tools/run_pipeline.sh --frames 200 --run-sweep

# Advanced run with TorchVision ResNet50, Memory Profiling, and a Live Desktop Output viewer!
./tools/run_pipeline.sh --frames 400 --resnet50 --memory --display --sched rr_workers --run-profiling
```

### 🖥️ Built-In GUI Launcher (Recommended)
If you prefer a visual interface to rapidly sweep architectures without typing complex bash commands, you can use the built-in desktop app:
```bash
python3 tools/gui_launcher.py
```
This requires `pip install customtkinter` and provides a tabbed UI to configure and run the entire pipeline seamlessly. It natively pipes standard output directly into its own embedded console.

#### GUI Configuration Tabs

**Tab 1: Simulation & Engine**
- **CARLA Host / Port**: Define the connection targeting your CARLA simulator instance.
- **Total Frames**: The number of simulation steps to run before terminating.
- **Target FPS**: The simulated camera capture rate (synchronizes with CARLA's `fixed_delta_seconds`).
- **TensorRT Engine**: Select from built-in compiled engines (e.g., `models/dummy.engine`, `models/fcn_resnet50.engine`) or provide a custom `.engine` absolute path.

**Tab 2: StarPU & Analyzers**
- **StarPU Parameters**:
  - **Scheduler**: Choose from 14 native algorithms (e.g., `dmda`, `ws`, `eager`, `heteroprio`).
  - **Run Scheduler Sweep**: Automatically iterates profiling runs across 4 core schedulers (`dmda`, `rr_workers`, `ws`, `eager`).
  - **In-flight tasks**: Controls the pipeline's maximum asynchronous task depth.
  - **CPU Workers**: Explicitly constrain StarPU's multi-core host mapping.
- **Trace Analyzers**:
  - **Record Memory / Bus Stats**: Logs Data Transfer and RAM allocations telemetry.
  - **APEX Profiling Mode**: Deep runtime structural tracing (`all`, `gtrace`, `gtrace-tasks`, `taskgraph`).

**Tab 3: Camera & Output**
- **Resolution & Format**:
  - **Input W / H**: Base focal resolutions captured directly from CARLA.
  - **Model W / H**: Downscaled NCHW array dimensions fed sequentially into TensorRT.
  - **Force --assume_bgra 1**: Architectural optimization bypassing slower RGBA extraction memory-logic.
- **Display / Dataset**:
  - **Live SDL2 Semantic Viewer**: Renders the 8-camera 360-degree composite inference loop directly on your host screen.
  - **Skip Inference**: Disables TensorRT prediction (`--no_pred`) generating pure ground-truth semantic mask outputs exclusively.

#### Execution Controls (Always visible bottom strip)
- **Run Dataset Extraction Mode**: Generates strictly synchronized pure images into `runs/` bypassing StarPU structural overhead.
- **Multi-Process Splitting**: Wraps the execution to separate client and server CPU/GPU telemetry tracking.
- **Preview Command**: Prints the exact underlying `./tools/run_pipeline.sh` execution string to the GUI console.
- **Run Pipeline**: Executes the configuration securely on an isolated background thread to prevent UI freezing.
- **Force Stop**: Sends explicit `pkill -9` sequences systematically resolving hanging UE4 engines or C++ binaries.

**Underlying Bash Overrides (`run_pipeline.sh`):**
- `--frames N`: Modify execution length.
- `--engine PATH` / `--deeplabv3` / `--resnet50`: Swap loaded models inline.
- `--memory` / `--display`: Switch live evaluation triggers securely.
- `--apex-mode MODE`: Wrap execution natively in `apex_exec`.
- Execution selection flags: `--run-dataset`, `--run-sweep`, `--run-split`, `--run-profiling`.

## 3. The Main ADV Pipeline

The pipeline connects to CARLA, extracts high-frequency RGB and semantic Ground Truth frames, preprocesses them (BGRA to NCHW), runs TensorRT inference, and calculates metrics (mIoU, Pixel Accuracy) asynchronously.

### Automated APEX Tracing & Verification Demo
The recommended way to test the StarPU framework is using the unified verification script. This script automatically launches CARLA, generates datasets or traces, and guarantees server teardown even on failures.

```bash
# Basic sweep through all schedulers
./tools/run_pipeline.sh --frames 200 --run-sweep

# Advanced run with APEX Tracing, Memory Bus profiling, and a TorchVision DeepLabV3 engine!
./tools/run_pipeline.sh --frames 400 --deeplabv3 --memory --apex-mode taskgraph --display --run-profiling --sched dmda
```

**Supported Options (`run_pipeline.sh` APEX flags):**
- `--apex-mode MODE`: Wrap execution natively in `apex_exec` (`taskgraph`, `gtrace`, `all`)
- `--trace-root PATH`: Redirect where traces are stored natively from `/tmp/starpu_traces_adv`.

### Live Semantic Visualization
The `pipeline_starpu` binary now natively supports a decoupled, lock-free **SDL2 Real-Time Viewer**! By passing `--display 1`, an external window will render the inference outputs. 
*Note:* TorchVision models (`--deeplabv3`, `--resnet50`) output 21 Pascal VOC classifications. The C++ pipeline natively detects this shape and securely intercepts VOC classes (e.g. `Class 7: Car`) into corresponding CARLA Cityscapes colors (e.g. `Class 10: Vehicle - Blue`) guaranteeing a stunning 25-color visualization without corrupting boundary mappings!

### StarPU Pipeline (Recommended)
This version uses StarPU to intelligently schedule CPU and GPU tasks.
```bash
./build/pipeline_starpu --host 127.0.0.1 --port 2000 --w 800 --h 600 --fps 20 --frames 200 --engine models/dummy.engine --out_w 512 --out_h 256 --inflight 2 --cpu_workers 8 --print_every 20 --eval_every 50
```

### Direct Pipeline (No StarPU)
A traditional `std::thread` based implementation for baseline comparisons.
```bash
./build/pipeline_direct --host 127.0.0.1 --port 2000 --w 800 --h 600 --fps 20 --frames 200 --engine models/dummy.engine --out_w 512 --out_h 256 --assume_bgra 1 --resize bilinear --print_every 20 --eval_every 50
```

**Outputs:**
- Both applications will generate a `pipeline_*.csv` in the `runs/` directory detailing per-frame stage latencies.

## 4. Performance Profiling & Workload Split Report

For hardware-characterization research, you can run isolated CPU core tests and automatically summarize telemetry across the CARLA server and the ADV client. 

**Run the Profiled End-to-End Suite:**
```bash
# 1. Launch Server and Client bound to specific cores:
SERVER_CORES="0-9" CLIENT_CORES="10-19" ./tools/run_profiled_single_machine.sh --port 2000 \
  --client ./build/pipeline_starpu \
  --client_args "--host 127.0.0.1 --port 2000 --w 800 --h 600 --fps 20 --frames 120 --engine models/dummy.engine --out_w 512 --out_h 256 --inflight 2 --cpu_workers 8"

# 2. In a second terminal, capture telemetry (run simultaneously):
./tools/monitor_split.sh --duration 60

# 3. When finished, generate the JSON/TXT summary:
python3 ./tools/summarize_run.py

# 4. Cleanup
./tools/kill_server.sh
```

**Outputs:** `runs/<timestamp>/summary.txt` cleanly separates Server CPU% from Client CPU%, along with VRAM utilization and StarPU End-to-End Latencies.

## 7. Troubleshooting Guide

- **`Spawn failed because of invalid actor description`**: Your map spawn points are occupied (a crashed car exists from a previous run), or `vehicle.mercedes.coupe_2020` is explicitly unavailable in your simulator's `.egg` distribution.
- **Client or Server PIDs tracking 0% CPU**: Because bash wrapper scripts (like `CarlaUE4.sh`) spawn children executing dense logic, the monitored PID might be anchored to a sleeping parent. The monitor tool tracks process trees natively to mitigate this, but `tail -f runs/pids.env` helps confirm valid active IDs.
- **Float Locale Mismatches**: On European systems, `pidstat` typically yields float values using commas (e.g., `3,40`). `summarize_run.py` sanitizes these automatically.
- **NaN Preprocess Outputs**: Standard deviation variants `[0.0, 0.0, 0.0]` were supplied to the normalizer, or tensor dimensions are physically zero.
- **Low Match Rate (FrameSync)**: Ensure `--sync 1` is applied. In asynchronous server execution, match rates drop to 60-70% due to CARLA phase jitter.
- **CUDA Out of Memory**: VRAM fragmentation from simultaneous headless CARLA rendering and deep generic TensorRT contexts. Decrease `w` and `h` resolution or limit `--inflight` tasks.
